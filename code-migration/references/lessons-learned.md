# 迁移经验教训

## 1. 迁移项目中不要提前引入设计模式

在代码迁移阶段，不要引入源工程没有的架构模式（pipeline、validator、DDD聚合根等）。迁移的首要目标是"跑通"，架构优化留到迁移完成后按需进行。已踩坑：提前建了 `pipeline/`、`split/`、`validator/`、`vo/` 占位包，后来全部删除。

## 2. 包结构先确认再建，避免反复移动文件

实体类从 `tradeorder/log/model/` 移到 `tradeorder/model/log/` 经历了两轮调整。根因是没有先把"技术层优先还是业务子域优先"这个问题想清楚就开始建包。正确做法：先在文档中画好目标结构，确认后再动手。

## 3. 源工程的命名习惯优先于"理论正确"

`dao` vs `mapper` vs `repository` 的选择：源工程用 `dao`，新工程一开始用了 `mapper`，后来改回 `dao`。迁移项目中，团队已有的命名习惯比理论上更"正确"的命名更重要，降低认知成本。

## 4. 渠道是入口，流程是主体

POP以销定采的代码应该放在 `retailrelation/`（以销定采流程）下，而不是 `tradeorder/pop/`（POP渠道）下。判断标准：代码跟着业务流程走，不跟着渠道入口走。否则每新增一个渠道就要在渠道包下复制一份流程代码。

## 5. 跨模块代码归属：谁的数据被写入，代码就在谁的模块里

分销出库生成PO单 → 代码在采购模块。取消DH单同步取消PO单 → 取消PO的逻辑在采购模块，分销模块只负责发起调用。不需要为跨模块协作建专门的包，通过 Service 间调用或 MQ 解决。

## 6. shared 和 common 的语义区分

`shared/` 和 `common/` 不能混用。`shared/` 只在工程顶层出现一次，放基础设施（config/feign/mq/util）和跨模块枚举。`common/` 在模块级别出现，只放模块内多个子域共享的枚举。子域级别直接用 `enums/`，不再嵌套 shared 或 common。已踩坑：一开始顶层同时有 `common/enums/` 和 `shared/`，语义重复导致枚举归属混乱。

## 7. 禁止用 PowerShell 批量处理含中文的 Java 文件

PowerShell 的 `Get-Content` + `Set-Content` 组合处理含中文注释的 Java 文件时，会导致 UTF-8 多字节字符被截断，产生乱码（如"待审核"变成"待审?"）。已踩坑：用 PowerShell 批量修改 package 声明时，26 个采购枚举文件和多个分销枚举文件的中文全部损坏。正确做法：用 `Write` 工具逐个重写文件，或者只用 PowerShell 做文件复制/移动/删除，不做内容修改。

## 8. 按接口迁移优于按层迁移

当源工程存在大 Service 类（几千行代码）时，按层迁移会导致跨模块依赖爆炸、重构延迟、进度不可见。正确做法：以接口/MQ 为迁移单元，边迁移边重构，每个接口独立验收。迁移前分析依赖，提前创建 Feign 接口桩处理跨模块调用。

## 9. 迁移时重构，而非迁移后重构

按层迁移时只做"搬运"不做重构，导致后续重构工作量巨大。正确做法：在迁移每个接口时同步重构——复用已迁移方法、优化命名、拆分大方法、提取常量。每个接口迁移完成后更新接口对照表，记录重构说明和待处理依赖。

## 10. 迁移开始前必须确认迁移范围

迁移开始阶段，必须由用户给出明确指示：迁移类型（接口/MQ/定时任务）、具体标识（类名、方法名、接口路径）。否则主动询问，避免迁移方向错误或遗漏依赖。已踩坑：未确认迁移范围就直接开始，导致遗漏字段变更检测逻辑。

## 11. 字段比对逻辑提取为工具类

当多个消费者/服务存在相似的字段比对逻辑时，应提取到 `shared/util/CompareUtil.java` 工具类中复用。工具方法包括：`anyChanged(Object... pairs)` 判断任意字段变更、`decimalChanged(BigDecimal, BigDecimal)` 处理 BigDecimal 比对（含 null 和精度处理）。

## 12. 类名应与监听的表名相关

DTS 消费者类名应与监听的表名相关，便于理解和维护。例如：监听 `purchase_psi_data` 表的消费者命名为 `PurchasePsiDataConsumer`，而非 `OrderBvsPurchaseConsumer`。

## 13. 迁移进度按链路维度标记，而非按技术类型

DTS 消费者 → MQ 生产者 → MQ 消费者 → 业务处理，是一条完整的数据流。把各个环节割裂到不同的迁移对照表（DTS 表、MQ 表分开），会导致"标记完成但链路不通"的错觉。正确做法：在迁移备忘录中按链路组织，每个环节单独标记状态，链路整体状态取最慢环节。阻塞点一目了然，便于多人协作和进度汇报。
