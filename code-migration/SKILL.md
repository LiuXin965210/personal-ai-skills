# 代码迁移 Skill

## 何时激活

- 执行代码迁移任务
- 讨论迁移方案或规范
- 决定代码/枚举的归属位置
- 处理跨模块依赖

## 激活后首要动作

1. 读取源工程和目标工程各自的 `README.md`，获取包结构规范、代码规范、模块划分
2. 读取 `~/.claude/skills/code-migration/references/migration-memo.md`，了解当前迁移进度
3. 读取 `~/.claude/skills/code-migration/references/lessons-learned.md`，了解已踩过的坑

## 迁移开始阶段（必须）

> **重要**：迁移开始前，必须由用户给出明确的迁移指示，否则主动询问。

### 必须确认的信息

| 信息项 | 说明 | 示例 |
|--------|------|------|
| **迁移类型** | 接口 / MQ / 定时任务 / 其他 | `迁移接口`、`迁移MQ消费者` |
| **具体标识** | 接口路径、类名、方法名 | `/api/order/create`、`OrderController.createOrder` |
| **源工程路径** | 源代码所在位置 | `ylh-cloud-service-order/src/main/java/...` |

### 标准启动模板

```
请确认以下迁移信息：
- 迁移类型：[接口/MQ/定时任务/其他]
- 具体标识：[接口路径或类名方法名]
- 源工程路径：[相对路径]

确认后开始迁移。
```

### 迁移类型与所需信息

| 迁移类型 | 必须提供的信息 |
|----------|----------------|
| **HTTP 接口** | 接口路径（如 `/api/order/create`）或 Controller 类名 + 方法名 |
| **MQ 消费者** | 消费者类名或监听的队列名/Topic |
| **MQ 生产者** | 生产者类名或目标队列名 |
| **定时任务** | XXL-Job Handler 类名或任务名称 |
| **DTS 消费者** | 消费者类名或监听的表名 |
| **实体类** | 表名或实体类名 |
| **枚举类** | 枚举类名 |

## 子智能体协作

迁移全流程中，按阶段调用以下子智能体：

| 阶段 | 子智能体 | 职责 |
|------|----------|------|
| 迁移前规划 | `architect` | 分析源工程代码结构，制定迁移计划，确认目标包结构 |
| 废弃代码识别 | `refactor-cleaner` | 识别源工程中无用/废弃/重复代码，标记跳过项 |
| 迁移后审查 | `code-reviewer` | 审查迁移后的代码质量、安全性、命名规范 |
| 数据库相关 | `database-reviewer` | 审查表结构设计、SQL 优化、跨模块 JOIN 拆分方案 |
| 单元测试 | `tdd-guide` | 指导 Service 层迁移后的单元测试编写 |
| 构建失败 | `build-error-resolver` | 迁移过程中编译失败时快速定位和修复 |

### 协作流程

```
architect 制定迁移计划（按接口优先级排序）
    → refactor-cleaner 标记废弃接口
    → 按接口迁移执行（边迁移边重构）
        ├── 分析源码依赖
        ├── 准备基础设施（Entity/Enum/Feign桩）
        ├── 迁移 Service + Controller
        ├── 编译验证
        └── 更新接口对照表
    → build-error-resolver 处理编译问题
    → code-reviewer 审查迁移代码
    → database-reviewer 审查 SQL/表结构变更（如有）
```

## 按接口迁移顺序

> **重要调整**：基于实际迁移经验，从"按层迁移"调整为"按接口迁移"。

迁移以**接口/MQ 为单元**进行，每个接口独立迁移和验收：

### 迁移优先级

| 优先级 | 判断标准 |
|--------|----------|
| P0 - 核心 | 无跨模块依赖、被高频调用、核心业务流程 |
| P1 - 重要 | 少量跨模块依赖（可提前处理）、重要业务流程 |
| P2 - 一般 | 有跨模块依赖需协调、非核心业务 |
| P3 - 低 | 疑似废弃、调用极少、可延后处理 |
| P4 - 跳过 | 已废弃、无实际调用 |

### 单接口迁移步骤

```
1. 分析源码
   - 定位 Controller 方法
   - 分析依赖的 Service/DAO/DTO/Enum
   - 标记跨模块依赖

2. 准备依赖
   - 迁移未迁移的 Entity/Enum（按需）
   - 创建跨模块 Feign 接口桩
   - 复用已迁移的公共方法

3. 迁移实现（边迁移边重构）
   - 迁移/重构 Service 方法
   - 迁移 Controller 方法
   - 编译验证

4. 记录结果
   - 更新接口对照表
   - 更新迁移备忘录
   - 标记待处理的跨模块依赖
```

### 迁移时重构规则

迁移接口时，同步进行以下重构：

| 重构类型 | 说明 |
|----------|------|
| 复用已迁移方法 | 优先调用已迁移的 DAO/Service 方法 |
| 优化变量命名 | 遵循驼峰命名，避免拼音缩写 |
| 拆分大方法 | 超过 50 行的方法拆分为多个私有方法 |
| 提取常量 | 魔法值提取为常量或枚举 |
| 简化条件 | 复杂 if-else 使用策略模式或提前返回 |
| 提取工具类 | 重复逻辑提取到 `shared/util/` |

## 核心规则

### 跨模块依赖处理

- JOIN 查询 → 拆分为 Feign 接口调用
- 跨模块事务 → 本地事务 + MQ 最终一致性
- 代码归属 → 谁的数据被写入，代码就在谁的模块里
- 发现跨模块依赖时暂停迁移，提示用户确认处理方式

### 代码合并规则

- 识别重复逻辑后提取公共方法，不照搬源代码
- 仅子域内使用 → `{subdomain}/service/` 私有方法
- 模块内多子域使用 → `{module}/common/` 公共类
- 跨模块使用 → `shared/util/`

### 枚举三级归属

| 层级 | 位置 | 判断标准 |
|------|------|---------|
| 全局 | `shared/enums/` | 被多个模块引用 |
| 模块 | `{module}/common/enums/` | 被同一模块内多个子域引用 |
| 子域 | `{subdomain}/enums/` | 仅被一个子域使用 |

### DAO 层共存

- JPA 放 `dao/jpa/`，MyBatis 放 `dao/mybatis/`，物理隔离不混放
- MyBatis XML 放 `resources/mybatis/mapper/{module}/{subdomain}/`
- Redis 操作放 `dao/redis/`
- 同一张表可同时有 Repository 和 Mapper

## 经验教训（强制遵循）

1. **不要提前引入设计模式** — 迁移阶段首要目标是"跑通"，架构优化留到迁移完成后
2. **包结构先确认再建** — 对照 README.md 中的规范确认后再动手，避免反复移动文件
3. **源工程命名习惯优先** — 团队已有的命名比"理论正确"更重要
4. **渠道是入口，流程是主体** — 代码跟着业务流程走，不跟着渠道入口走
5. **谁的数据被写入，代码就在谁的模块里** — 跨模块代码归属核心原则
6. **shared 和 common 语义区分** — shared 只在顶层，common 在模块级别
7. **禁止用 PowerShell 批量处理含中文的 Java 文件** — 使用 Write 工具逐个写入

## 迁移检查清单

### 单接口完成后
- [ ] 源接口和目标接口路径已记录到对照表
- [ ] 代码放到正确包路径
- [ ] 命名与源工程一致（或已优化）
- [ ] 编译通过（`mvn compile`）
- [ ] 跨模块依赖已标记或已处理
- [ ] 无 TODO 标记残留（或已记录到待处理列表）
- [ ] 迁移备忘录已更新

### 重构检查（迁移时同步完成）
- [ ] 已复用已迁移的 DAO/Service 方法
- [ ] 变量命名符合驼峰规范
- [ ] 超过 50 行的方法已拆分
- [ ] 魔法值已提取为常量或枚举
- [ ] 复杂条件逻辑已简化
- [ ] 重复逻辑已提取为工具类

### 模块完成后
- [ ] 模块内 P0 接口全部迁移完成
- [ ] 模块内 P1 接口全部迁移完成
- [ ] `mvn compile` + `mvn test` 全部通过
- [ ] 迁移备忘录已更新
- [ ] 无源工程旧包名残留
- [ ] `code-reviewer` 审查通过
