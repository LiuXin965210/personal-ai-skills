# 需求设计稿生成

## 工作流程

1. 判断用户是否提供模板。多数情况下用户不会提供模板；未提供模板时，直接使用内置默认模板 `references/template.md`，不要停下来索要模板。
2. 若用户明确提供模板，读取用户模板。模板是 PDF 时优先提取 PDF 文本和章节结构；模板是 Markdown 或文本时直接解析标题、表格和示例字段。用户模板与默认模板冲突时，以用户本次模板为准。
3. 读取用户提供的需求设计稿。若是 URL，先打开并提取正文；若因登录、权限、SSO、VPN、网络或反爬限制无法读取，使用 in-app Browser / computer use 打开链接，请用户在浏览器内完成登录或授权，不要在聊天中索要密码、验证码、cookie 或 token。
4. 若用户提供后端工程，结合工程代码识别接口、Service、Consumer、配置项、数据模型、SQL、枚举、日志和外部依赖；若未提供工程，仅基于需求内容产出设计稿，并把无法确认的实现细节标记为 `待确认` 或 `建议`。
5. 按模板格式生成 Markdown 设计稿。默认模板始终可用；只在需要确认章节格式时读取 `references/template.md`。
6. 默认将生成的 Markdown 设计稿保存到当前工作区根目录，也就是当前 `cwd`。文件名使用一级标题去掉 `#` 后的文本，例如 `B端权益-直发门店增加物流信息后端设计.md`。若用户明确指定保存路径或文件名，以用户指定为准。
7. 最终回复只给已保存文件的路径和必要说明，除非用户要求直接展示全文。不要输出与设计稿无关的寒暄、分析草稿或工具日志。

## 输出要求

- 使用中文输出。
- 一级标题使用需求名称加设计稿类型，例如 `# 净水零售托盘商以销定采后端设计`。
- 未提供模板时，按默认模板保留主要章节：变更记录、需求链接、外围相关接口、接口设计、Kafka 设计、Service 设计、数据模型、增量变更脚本、待确认事项。
- 提供模板时，按用户模板保留章节；默认模板只作为补充参考。
- 缺失或不涉及的章节不要删除，写 `本期无调整`、`暂无` 或 `待确认`。
- 接口、字段、SQL、类名、方法名、配置项、Kafka topic、枚举值使用 Markdown 代码格式。
- 表格优先使用 Markdown 表格。
- 对无法从需求或工程确认的内容，不要编造确定结论；用 `待确认` 标注。
- 若发现模板 PDF 中有明显格式章节，而默认模板未覆盖，应补充到输出中。
- 默认把设计稿写入当前工作区根目录的 `.md` 文件；文件内容必须是完整 Markdown 设计稿正文。

## 后端工程处理

当用户提供后端工程名称、路径或仓库信息时：

1. 先定位工程目录和相关模块。
2. 用代码搜索确认已有接口、Service、Consumer、Mapper、Entity、DTO、配置项和 SQL。
3. 设计稿中的工程名、接口路径、方法名、类名、字段名必须尽量来自代码事实。
4. 若只能推断，明确写为 `建议新增`、`建议调整` 或 `待确认`。

## 参考模板

默认模板见 `references/template.md`。用户未提供模板时直接使用默认模板生成设计稿；不要要求用户额外提供模板。
