# Git Commit Helper

分析本地 Git 变更，识别 GitLab/Gerrit 工作流，按项目规范生成提交消息，并在需要时拆分为多个逻辑提交。

## 适用场景

- 提交当前工作区变更。
- 根据需求号生成规范的提交消息。
- 将配置、业务代码或不同需求拆成多次提交。
- 指定文件和提交消息执行提交。

## 提交消息格式

默认格式：

```text
{用户名}-{需求号} {类型} {变更内容}
```

常用类型包括 `feat`、`fix`、`chore`、`docs`、`refactor`、`test` 和 `perf`。

## 使用方式

自动分析并提交：

```text
使用 git-commit-helper 分析当前变更并提交，需求号是 yilihuo-51952
```

指定文件：

```text
提交 TradeOrder.java，消息为
yilihuo-51952 feat 新增分销订单实体类
```

指定分组：

```text
分两批提交：
1. pom.xml，消息为 yilihuo-51952 chore 更新依赖
2. 其余业务文件，消息为 yilihuo-51952 feat 新增采购功能
```

## 默认流程

1. 检查远程仓库、当前分支和工作区状态。
2. 识别 GitLab 或 Gerrit 工作流。
3. 分析 diff，生成提交分组和消息。
4. 展示计划并按用户确认执行。
5. 提交后返回 commit 信息；是否推送按用户要求执行。

## 注意事项

- 同一需求的强关联代码默认放在同一个提交中。
- 不会把无关的未跟踪文件自动加入提交。
- 遇到合并冲突会立即停止，由用户手动解决，不会自行选择 `ours` 或 `theirs`。
- 提交和推送是写操作，请明确说明是否只提交、是否推送以及目标分支。
