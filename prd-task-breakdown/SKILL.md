# PRD 任务拆解

## 工作流程

1. 读取用户提供的 PRD。若用户提供文本，直接提取需求内容；若用户提供 URL，先打开并提取文档正文。
2. 若 URL 因登录、权限、网络、SSO、VPN、反爬等原因无法直接读取，使用 computer use / Codex in-app Browser 打开链接，请用户在浏览器中完成登录或授权。用户确认可访问后，从当前页面提取可见文本；文本不足时再结合截图识别。若浏览器仍无法访问，请用户粘贴正文或提供导出的 Markdown、Word、PDF。
3. 识别需求背景、功能范围、页面变更、接口/数据取数、权限、导入导出、异常场景、边界规则和测试范围。
4. 按用户指定的后端、前端、测试工时预算拆分。若缺少任一角色预算，先说明缺失项并要求补充，再输出最终拆解。
5. 将任务项合并成连续序号列表。不要按模块分段展示；不要输出多个独立卡片式段落。
6. 每个序号尽可能覆盖接近 5 天的工作量，但单项合计不得超过 5 天。
7. 默认每个序号只输出任务项本身，不展示该序号对应的后端、前端、测试工时；只在总计行展示角色工时汇总。
8. 只有用户明确要求“逐项展示工时”或“每项标注角色工时”时，才在每个序号后追加 `后端X天 前端Y天 测试Z天 合计N天`。
9. 输出前检查所有硬性约束。

## URL 访问兜底

当 PRD 链接无法通过常规方式读取时：

1. 先用 in-app Browser / computer use 打开链接，不要立即停止。
2. 明确告诉用户当前阻塞原因，例如登录页、无权限、需要 VPN、页面空白或页面报错。
3. 请用户只在浏览器中完成登录、SSO、VPN 或权限确认，不要在聊天中索要密码、验证码、cookie 或 token。
4. 用户确认页面可访问后，读取浏览器当前页面文本；必要时使用截图辅助识别表格、标题和原型说明。
5. 若用户协助后仍无法访问 PRD，请用户粘贴 PRD 正文或导出文档。

## 硬性约束

- 单个序号任务项合计工时不得超过用户指定的单项上限；若未指定，默认按 5 天以内规划。
- 所有任务项合计工时不得超过 40 天。
- 每个任务项描述控制在 15-20 个中文字符内，不包含序号和工时文字。
- 默认不要在每个任务项后展示后端、前端、测试工时。
- 总计行必须展示后端、前端、测试工时汇总；某角色无工作量时写 `0天`。
- 后端、前端、测试总计必须分别匹配用户给定预算。若 PRD 范围无法匹配预算或超过 40 天上限，说明冲突并给出最接近的可行拆分。
- 不要输出笼统任务项，例如 `其他处理`、`杂项优化`、`联调支持`。除非 PRD 中有明确对应交付物。

## 输出格式

默认使用中文。输出连续序号列表，每个序号是一个任务项；不要使用模块标题分段，不要在任务项下继续列细分点，默认不要在每个序号后展示角色工时。

```markdown
1. 任务描述

2. 任务描述

总计：后端X天 前端Y天 测试Z天 合计N天
```

当用户明确要求逐项工时时，才使用：

```markdown
1. 任务描述：后端X天 前端Y天 测试Z天 合计N天
2. 任务描述：后端X天 前端Y天 测试Z天 合计N天
总计：后端X天 前端Y天 测试Z天 合计N天
```

## 拆分原则

- 优先把相关能力合并到同一个序号中，让每个序号尽量接近 5 天但不超过 5 天。
- 不要把同一条业务链路拆成多个分段标题。应合并为连续序号，例如 `1. 物流展示`、`2. 物流取数`、`3. 物流导出`、`4. 导入限制`。
- 如果单个能力超过 5 天，再按数据模型/API、页面交互、状态规则、导入导出、异常场景、回归测试等可交付边界拆分。
- 每个序号只保留一条任务项描述，不要在下方展开 2-6 条具体任务点。
- 可以出现后端-only、前端-only、测试-heavy 任务项，但默认只在总计行体现角色工时分布。
- 任务描述示例：
  - `直发门店物流信息展示规则调整功能`
  - `直发门店物流状态取数规则处理功能`
  - `多物流单号异常状态展示处理逻辑`
  - `直发门店物流信息导出字段支持功能`
  - `直发门店导入字段长度限制处理功能`

## 工时分配规则

- 后端工时包括接口设计、数据模型、业务规则、取数逻辑、集成、定时任务、权限、日志、迁移和导出能力。
- 前端工时包括页面字段、表单、列表列、状态展示、权限控制、交互状态、错误提示和导入模板说明。
- 测试工时包括用例设计、功能验证、回归、边界场景、权限场景、导入导出校验和集成验证。
- 必要时可使用 `0.5天`，否则优先使用整数天。
- 若用户给出角色级总预算，必须在内部完成任务项工时分摊，并让总计行与预算完全一致。默认不输出内部逐项分摊。

## 输出前校验

输出前手动检查：

1. 每个序号合计不超过 5 天。
2. 总工时不超过 40 天。
3. 默认每个序号后没有角色工时；除非用户明确要求逐项工时。
4. 每个任务描述保持 15-20 个中文字符。
5. 总计行的后端、前端、测试工时分别匹配用户给定预算。
6. 输出是连续序号列表，不是按模块分段的多段卡片。
7. 每个序号下面没有细分子任务点。

也可以将结果保存为 Markdown 后运行：

```bash
python3 scripts/validate_breakdown.py path/to/output.md --backend X --frontend Y --test Z
```

若用户要求逐项展示工时，再增加 `--require-item-effort`。
